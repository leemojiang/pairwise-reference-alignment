# 逐一修复时的 Git 使用流程

当接收到“逐一解决”“逐一修复”“每解决一个问题 commit 一次”等类似命令时，按如下规范使用 Git。

## 基本命令

- 使用 `git status --short` 查看当前工作区状态。
- 使用 `git add <file>` 只暂存本次问题相关的文件。
- 使用 `git commit -m "commit message"` 提交已经验证通过的单个问题。

## 执行规则

1. 开始前先运行 `git status --short`，确认当前工作区里是否已有 staged、unstaged 或 untracked 文件。
2. 如果已有 staged changes，先不要直接提交。需要判断这些 staged 文件是否属于当前任务：
   - 如果属于当前任务，可以继续处理。
   - 如果不属于当前任务，不要把它们混入后续 commit；必要时先提示用户处理。
3. 如果已有 unstaged 或 untracked 文件，不要默认修改、删除或提交它们。只有当这些文件确实属于当前问题时，才可以纳入对应 commit。
4. 一个问题对应一个 commit。每解决完一个问题后，先运行必要测试或构建检查；通过后，只 stage 这个问题相关的文件，然后生成一个 commit。
5. commit message 由执行者自动生成，应该简短描述该问题的实际修复内容，例如 `fix: group selected material files correctly`。
6. 如果某个问题的测试无法运行，或者测试失败但问题已经尽力修复，需要先说明原因，不要伪造“测试通过”。
7. 在所有问题完成前不要停止。完成一个问题并 commit 后，继续处理下一个问题。
8. 如果处理过程中发现其他人修改了文件，或出现和当前问题无关的新文件，只保留并避开它们，不要回滚、删除或提交。

9. 如果有的问题后面有用括号的备注,请遵循这些备注,举个例子:
- `(optional)`，表示这个问题是可选的，可以先跳过，等其他问题都完成后再回头处理它。
- `(和我讨论/不确定/需求困难)`，表示这个问题中间可能有不确定的地方,需要和我讨论后才能确定如何处理，可以先跳过，等其他问题都完成后再回头处理它。处理的时候有不确定的随时进行讨论而不是强行解决.


## 推荐工作循环

对每一个问题重复以下步骤：

1. 阅读相关代码并定位问题。
2. 修改代码。
3. 运行针对性测试；必要时运行完整测试或构建。
4. 使用 `git status --short` 检查变更范围。
5. 使用 `git add <file>` 只暂存当前问题相关文件。
6. 使用 `git commit -m "..."` 提交当前问题。
7. 继续处理下一个问题。
